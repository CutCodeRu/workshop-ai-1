# 07. Stack Research

Актуально на **3 марта 2026**.

## Контекст задачи

Нужно сделать Telegram-бота для малого бизнеса, который:

- отвечает на вопросы клиентов по базе знаний компании;
- собирает заявки: `имя`, `телефон`, `вопрос`;
- уведомляет владельца о новых заявках;
- может стартовать без публичного сервера, в режиме локального запуска.

Для этого выбран подход **RAG (Retrieval-Augmented Generation)**, несмотря на то, что для части задач он избыточен. Решение осознанное: нужна расширяемая архитектура под базу знаний и более качественные ответы по внутренним материалам компании.

## Целевой архитектурный подход

На старте целесообразен **классический 2-step RAG pipeline**:

1. Получить сообщение из Telegram.
2. Определить интент: обычный вопрос или заявка.
3. Для вопроса:
   - найти релевантные фрагменты в базе знаний;
   - собрать контекст;
   - передать контекст и вопрос в LLM;
   - вернуть ответ пользователю.
4. Для заявки:
   - провести пользователя по сценарию сбора данных;
   - сохранить заявку;
   - отправить уведомление владельцу.

Почему именно так:

- проще, чем agentic-подход;
- дешевле в сопровождении;
- предсказуемее в отладке;
- лучше подходит команде с разным уровнем опыта.

## 1. Язык и фреймворк для Telegram-бота

### Варианты

| Вариант | Плюсы | Минусы | Сложность освоения |
|---|---|---|---|
| **Python + aiogram 3** | Сильный стек для Telegram, async-first, FSM/Scenes, удобен для интеграции с AI/RAG в Python | Нужно понимать `asyncio`, middleware и DI-подход aiogram 3 | Средняя |
| **TypeScript + grammY** | Отличный DX, типизация, современный Node-стек | При RAG часто все равно появляется Python-часть или отдельный AI-сервис | Средняя |
| **TypeScript + Telegraf** | Простой вход, много примеров | Экосистема под сложные сценарии и современный Bot API обычно менее удобна, чем у grammY | Низкая-Средняя |
| **Go + telebot / tgbotapi** | Производительность, простота деплоя, хорош для надежного продакшена | Выше порог входа, меньше готовых материалов по RAG-интеграции | Средняя-Высокая |

### Вывод

Для проекта с RAG-базой знаний наиболее рационален **Python + aiogram 3**:

- Python уже является стандартной средой для embeddings, retrieval и LLM-интеграций;
- не нужно делить систему между разными языками;
- aiogram 3 хорошо подходит для сценариев с FSM, административными уведомлениями и расширением логики.

## 2. Хранение и поиск по базе знаний

### Что важно в 2025-2026

Для базы знаний компании обычно недостаточно только vector search. На практике лучше всего работают:

- **vector search** для смысловой близости;
- **full-text / keyword search** для точных терминов, тарифов, артикулов и формулировок;
- **гибридный retrieval** как основной рабочий вариант.

### Варианты

| Вариант | Плюсы | Минусы | Сложность освоения |
|---|---|---|---|
| **PostgreSQL + pgvector + Full Text Search** | Один datastore для заявок, пользователей и знаний; гибкие SQL-фильтры; можно собрать hybrid search | Retrieval-логику нужно продумать и реализовать самостоятельно | Средняя |
| **Qdrant** | Сильный специализированный retrieval-движок; хороший UX для векторного и гибридного поиска | Обычно отдельный сервис, отдельная инфраструктура | Средняя |
| **Chroma** | Очень быстрый старт локально, удобно для MVP | Часто перерастается и заменяется в production | Низкая |
| **Elasticsearch / OpenSearch** | Очень сильный поиск, гибрид, фильтрация, объяснимая релевантность | Тяжелее всего в сопровождении и инфраструктуре | Высокая |
| **Weaviate** | Богатый AI-native функционал | Часто избыточен для малого бизнеса на старте | Средняя-Высокая |
| **Hosted File Search** | Минимум кода, быстрый старт | Vendor lock-in, меньше контроля над retrieval-логикой | Низкая |

### Вывод

Для данного проекта выбран **PostgreSQL + pgvector**, дополненный **PostgreSQL Full Text Search**.

Почему это хороший выбор:

- один database stack для всей системы;
- проще операционно, чем отдельная векторная БД;
- удобно хранить рядом:
  - документы базы знаний;
  - чанки;
  - embeddings;
  - заявки клиентов;
  - историю обращений;
- можно построить собственный **hybrid retrieval**:
  - сначала full-text и vector search;
  - затем merge / rerank;
  - затем отбор top-k.

Это немного сложнее, чем взять готовую векторную БД, но лучше контролируется и лучше соответствует выбранному подходу: **самописный pipeline**.

## 3. LLM для генерации ответов

### Варианты

| Вариант | Плюсы | Минусы | Сложность освоения |
|---|---|---|---|
| **OpenAI** | Сильное качество, зрелая экосистема, хорошие embeddings и инструменты | Облачная зависимость, регулярные расходы | Низкая-Средняя |
| **Anthropic Claude** | Сильные ответы и работа с длинным контекстом | Стоимость, чуть менее прямолинейный стек вокруг retrieval для MVP | Низкая-Средняя |
| **Google Gemini** | Большой контекст, хорошее соотношение цена/качество | Поведение экосистемы и API может меняться быстрее, чем хочется для спокойного backend-ритма | Низкая-Средняя |
| **Ollama + локальные модели** | Приватность, локальный запуск, автономность | Требует железа, сложнее обеспечить стабильное качество и latency | Средняя-Высокая |

### Практический вывод

На старте для малого бизнеса разумнее брать **облачную LLM**, а не локальную модель:

- лучше качество ответов;
- меньше времени на инфраструктуру;
- быстрее запуск MVP;
- проще поддержка для mixed-skill команды.

При этом архитектуру надо строить так, чтобы LLM-провайдер был заменяемым:

- отдельный адаптер провайдера;
- единый интерфейс `generate_answer(...)`;
- отдельный интерфейс для embeddings.

### Рекомендуемая стратегия

- стартовать с **OpenAI** или **Gemini**;
- не привязывать бизнес-логику к SDK конкретного вендора;
- предусмотреть в будущем возможность переключения на локальные модели через Ollama.

## 4. Оркестрация RAG pipeline

### Варианты

| Вариант | Плюсы | Минусы | Сложность освоения |
|---|---|---|---|
| **LlamaIndex** | Очень хорош именно под RAG и ingestion документов | Добавляет собственные абстракции и слой фреймворка | Средняя |
| **LangChain / LangGraph** | Огромная экосистема, множество интеграций | Для простого FAQ/RAG может быть избыточен | Средняя |
| **Haystack** | Production-minded pipeline-подход | Более “инженерный” и менее легкий старт | Средняя |
| **Flowise** | Быстрые визуальные прототипы | Ограничения при сложной кастомной логике | Низкая-Средняя |
| **Самописный pipeline** | Полный контроль, простая ментальная модель, нет лишней магии | Все нужно продумать самостоятельно: ingestion, retrieval, ranking, tracing, fallback | Средняя-Высокая |

### Вывод

Для этого проекта выбран **самописный pipeline**.

Причины выбора:

- нужен понятный и контролируемый backend, а не фреймворк поверх фреймворка;
- retrieval будет завязан на `PostgreSQL + pgvector + FTS`, что удобно реализовать вручную;
- у команды будет прозрачное понимание каждого этапа:
  - chunking;
  - embeddings;
  - сохранение в БД;
  - hybrid retrieval;
  - prompt assembly;
  - answer generation;
  - fallback и логирование.

Это особенно полезно, если проект нужно будет постепенно доращивать без полной зависимости от внешней RAG-абстракции.

## Финальное решение

### Зафиксированный стек

- **Язык:** Python 3.12+
- **Telegram framework:** aiogram 3
- **База данных:** PostgreSQL
- **Векторный поиск:** pgvector
- **Текстовый поиск:** PostgreSQL Full Text Search
- **RAG orchestration:** самописный pipeline
- **LLM:** облачный провайдер через заменяемый адаптер
- **Embeddings:** отдельный провайдерный адаптер
- **Режим запуска на старте:** long polling без публичного сервера

### Почему выбран именно этот стек

#### 1. Он реалистичен для команды с разным уровнем опыта

- aiogram 3 закрывает Telegram-часть на зрелом и понятном инструменте;
- PostgreSQL знаком многим backend-разработчикам;
- самописный pipeline проще объяснить и дебажить, чем сложный orchestration framework;
- не нужно поддерживать сразу несколько языков и экосистем.

#### 2. Он достаточно простой для MVP, но не тупиковый

- можно быстро стартовать локально;
- можно позже вынести в Docker и VPS;
- можно масштабировать retrieval и дорабатывать ranking;
- можно заменить LLM-провайдера без переписывания ядра.

#### 3. Он хорошо подходит именно под knowledge bot

Для вашей задачи важнее:

- предсказуемость;
- контролируемый retrieval;
- прозрачные данные;
- управляемая стоимость;

чем “магия” сложных AI-агентов.

## Рекомендованный RAG-подход внутри выбранного стека

### Ingestion

1. Загружать документы из подготовленных источников:
   - markdown;
   - txt;
   - pdf после извлечения текста;
   - FAQ-таблиц;
   - внутренних регламентов.
2. Нормализовать текст.
3. Разбивать на чанки фиксированного или адаптивного размера.
4. Сохранять:
   - документ;
   - chunk;
   - метаданные;
   - embedding.

### Retrieval

1. Выполнять full-text поиск по `tsvector`.
2. Выполнять vector search через `pgvector`.
3. Объединять результаты.
4. Применять простой rerank:
   - по суммарному score;
   - по совпадению категорий;
   - по свежести или приоритету документа.
5. Брать top-k фрагментов в prompt.

### Generation

1. Собирать системный prompt с правилами:
   - отвечать только по найденному контексту;
   - не выдумывать факты;
   - если информации недостаточно, предлагать оставить заявку.
2. Передавать:
   - вопрос пользователя;
   - найденные фрагменты;
   - краткие metadata citations при необходимости.
3. Возвращать краткий и безопасный ответ.

### Fallback-поведение

Если retrieval не нашел уверенного ответа:

- не “галлюцинировать”;
- сообщать, что точной информации нет;
- предлагать оставить заявку;
- передавать владельцу вопрос клиента.

## Рекомендации по границам MVP

### В MVP стоит включить

- Telegram-бот на aiogram 3;
- сценарий FAQ-вопросов;
- сценарий создания заявки;
- PostgreSQL;
- `pgvector`;
- таблицы документов и чанков;
- минимальный ingestion из markdown/txt;
- hybrid retrieval;
- один LLM-провайдер;
- логирование запросов и ответов.

### В MVP можно не включать

- агентные сценарии;
- сложный reranker;
- мультиязычный retrieval;
- web-admin;
- автоматическую переиндексацию большого числа источников;
- сложную observability-платформу.

## Краткий итог

Для этого проекта принято следующее решение:

**`aiogram 3 + PostgreSQL + pgvector + PostgreSQL FTS + самописный RAG pipeline`**

Это не самый “магический” стек, но он:

- понятный;
- расширяемый;
- инженерно контролируемый;
- подходит для локального старта;
- хорошо переносится в production;
- удобен для команды с разным уровнем опыта.
